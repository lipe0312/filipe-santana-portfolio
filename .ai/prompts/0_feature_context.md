# Portfolio Feature Context

## 1. Feature Name & Intent

**Feature Name:** Advanced Motion Layer â€” Global Scroll Reveals, Quantitative Counter, Magnetic Tags & Precision Cursor Glow
**Business/UX Value:** This milestone is the difference between a portfolio that looks good and one that _feels_ alive. The scroll reveals reward the user for reading, creating a sense of progressive disclosure that makes the content feel intentional rather than dumped onto the page. The quantitative counter transforms static numbers into earned moments of impact. The magnetic tag repulsion creates a subliminal sense that the UI is aware of the user â€” a signature interaction that is rare enough to be genuinely memorable without ever calling attention to itself. The cursor glow, adapted for Light Mode as a soft shadow or tint, makes the Bento Grid feel like a physical surface responding to presence. Together, these four effects operate at the threshold of conscious perception, which is precisely where the most effective UI motion lives. This is the Linear.app touch: you feel it more than you see it.
**Priority:** High
**Explanation:** Every effect in this milestone must be implemented using native browser APIs and React primitives only. No physics libraries, no heavy animation runtimes beyond what is already installed (Framer Motion, if present from Milestone 5). The performance contract is non-negotiable: all effects must sustain 60fps on mid-range hardware, produce zero Cumulative Layout Shift (CLS), and be completely invisible to users who have enabled `prefers-reduced-motion` or are on touch devices. These are not degraded experiences â€” they are the correct, complete experience for those contexts. An effect that tanks performance or forces layout recalculations is worse than no effect at all. Every `mousemove` listener must be throttled through `requestAnimationFrame`. Every `IntersectionObserver` must disconnect after firing. Every effect must be cleaned up in its `useEffect` return function without exception.

The counter triggers only when the containing element enters the viewport, detected by the same `IntersectionObserver` infrastructure used for scroll reveals. The observer disconnects after the counter fires â€” it must not re-trigger on subsequent scroll passes.

- [x] **`02_design_assets/brand_system/functional-specification.md` â€” Â§6, Innovation Â§3 (Magnetic Tag Repulsion):** Defines the repulsion as a `mousemove` handler computing the distance between the cursor and each tag's center via `getBoundingClientRect()`. When the cursor is within `60px` of a tag's center, the tag shifts its `transform: translate()` away from the cursor by a magnitude proportional to proximity, capped at `6px` in any direction. On `mouseleave`, the tag's transform resets to `translate(0, 0)` with an elastic snap-back (implemented as a CSS `transition` with a slight `ease-out` or `cubic-bezier` that slightly overshoots before settling â€” maximum overshoot `1px` to remain within the "restrained" aesthetic). Must be disabled on `hover: none` and `prefers-reduced-motion: reduce`.

- [x] **Performance & Accessibility â€” Mandatory System Constraint:** Every `mousemove` listener in this milestone (cursor glow, magnetic tags) must be throttled through a `requestAnimationFrame` gate: set a boolean `rafPending` flag to `true` when a frame is scheduled, call `requestAnimationFrame`, and reset the flag inside the frame callback before processing. This prevents listener callbacks from firing faster than the display refresh rate. Every `useEffect` that registers a listener or observer must return a cleanup function that removes the listener and/or calls `observer.disconnect()`. Failure to clean up is a memory leak. All effects must check `window.matchMedia('(prefers-reduced-motion: reduce)').matches` and `window.matchMedia('(hover: none)').matches` before attaching any listener â€” if either condition is true, the effect must not attach.

## 4. External Documentation / Inspirations

> âš ď¸Ź **The implementing agent must open and read each of these URLs directly before writing any code. These references define the exact implementation patterns required â€” do not substitute alternative approaches.**

- **`https://developer.mozilla.org/en-US/docs/Web/API/Intersection_Observer_API`:** The definitive reference for `IntersectionObserver`. Study the `threshold` option, the `unobserve()` method (called immediately after the callback fires on each element to prevent re-triggering and memory accumulation), and the `rootMargin` option (not used here, but understand why â€” we want viewport-relative triggering, not offset triggering). Pay attention to the callback's `isIntersecting` property â€” only apply `.is-visible` when `isIntersecting === true`.

- **`https://developer.mozilla.org/en-US/docs/Web/API/window/requestAnimationFrame`:** The definitive reference for `requestAnimationFrame`. Study the frame callback timing, the returned handle (required to call `cancelAnimationFrame` in cleanup), and the pattern for building a throttled event listener using `rAF` as a gate. The counter animation `ease-out` curve must be implemented using the elapsed time ratio pattern: `progress = elapsed / duration`, then `easedProgress = 1 - Math.pow(1 - progress, 3)` (cubic ease-out), then `currentValue = Math.round(start + (end - start) * easedProgress)`.

- **`https://developer.mozilla.org/en-US/docs/Web/CSS/Using_CSS_custom_properties`:** Reference for the CSS custom property (`--x`, `--y`) pattern used by the cursor glow. Understand how `element.style.setProperty('--x', value)` updates a CSS variable from JavaScript and how `radial-gradient(200px circle at var(--x) var(--y), ...)` consumes it in a `background` declaration. This pattern avoids React re-renders entirely â€” the glow updates happen outside the React render cycle for maximum performance.

- **`https://developer.mozilla.org/en-US/docs/Web/API/Element/getBoundingClientRect`:** Reference for the `getBoundingClientRect()` call used in the magnetic tag repulsion to compute each tag's center coordinates (`rect.left + rect.width / 2`, `rect.top + rect.height / 2`) and the cursor-to-center distance vector (`dx = cursor.x - tagCenter.x`, `dy = cursor.y - tagCenter.y`). The repulsion translation vector is the inverse of this: `translateX = -dx * (strength / distance)`, `translateY = -dy * (strength / distance)`, clamped to `6px`.

- **`https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion`:** Reference for the `prefers-reduced-motion` media feature. The JavaScript check is `window.matchMedia('(prefers-reduced-motion: reduce)').matches`. Additionally, add a CSS rule in `globals.css` as a safety net: `@media (prefers-reduced-motion: reduce) { .reveal { transition: none; opacity: 1; transform: none; } }` â€” this ensures that even if the JavaScript observer fires late, reduced-motion users never see a flash of invisible content.

- **`https://developer.mozilla.org/en-US/docs/Web/CSS/@media/hover`:** Reference for the `@media (hover: none)` touch device detection. The JavaScript equivalent is `window.matchMedia('(hover: none)').matches`. Both the CSS and JS checks must be applied: CSS disables the static glow `div` visibility, JS prevents the listener from attaching.

- **`https://linear.app/`:** Motion benchmark reference. Study the speed and restraint of every transition on this page. Notice that nothing bounces, nothing overshoots more than a pixel, and nothing demands attention. The scroll reveals are fast and decisive. This is the feeling â€” not the implementation â€” that this milestone must achieve.

- **`https://web.dev/articles/cls`:** Reference for understanding Cumulative Layout Shift (CLS). Every effect in this milestone that modifies `transform` on an element must use `transform` and `opacity` only â€” never `top`, `left`, `width`, `height`, `margin`, or `padding` â€” because only `transform` and `opacity` are GPU-composited and CLS-free. The magnetic tag repulsion (`transform: translate()`) and the scroll reveals (`transform: translateY()`, `opacity`) are already compliant. Verify no effect in this milestone triggers a layout recalculation by reading or writing layout-affecting properties inside the `rAF` callback.

- **`https://web.dev/articles/rendering-performance`:** Reference for the `requestAnimationFrame` throttle gate pattern and the principle of keeping all JS work inside the animation frame budget (under 10ms per frame for 60fps). Study the "avoid layout thrash" section â€” reading `getBoundingClientRect()` inside a `mousemove` handler that also writes `style.transform` on multiple elements causes forced synchronous layout. The correct pattern: read all `getBoundingClientRect()` values in one pass, then write all `style.transform` values in a second pass, both inside the same `rAF` callback.

## 5. Acceptance Criteria

- [ ] All section components with the `.reveal` class (`Experience` entries, `About` blocks, `Contact`) transition from `opacity: 0, translateY(14px)` to `opacity: 1, translateY(0)` over `500ms ease-out` when they enter the viewport at `threshold: 0.12`. The `IntersectionObserver` calls `unobserve()` on each element immediately after the callback fires â€” no element is observed more than once.
- [ ] The stagger delay chain is correctly applied: eyebrow/label elements at `0ms`, primary headings at `50ms`, body copy at `100ms`, CTA buttons or tag rows at `150ms`. Elements beyond index 3 in a stagger chain share the `150ms` delay. All delays are applied as inline `transition-delay` values, not as `setTimeout` calls.
- [ ] A `@media (prefers-reduced-motion: reduce)` rule exists in `globals.css` that sets `.reveal` to `transition: none; opacity: 1; transform: none;`. Users with reduced motion enabled see all content immediately without any animation, and no content is ever invisible on initial render for these users.
- [ ] The quantitative counter in the About section animates from `0` to its final value over `1200ms` using a cubic ease-out curve (`1 - Math.pow(1 - progress, 3)`). The counter triggers only when the element enters the viewport via `IntersectionObserver` and does not re-trigger on subsequent scroll passes. The observer disconnects after the counter fires.
- [ ] The Precision Cursor Glow `div` is rendered with `pointer-events: none` and `position: fixed`, scoped visually to the Bento Grid section. It updates `--x` and `--y` CSS custom properties via a `mousemove` listener throttled through `requestAnimationFrame`. The gradient uses a Light Mode-appropriate tint (soft shadow, low-opacity warm or cool tone â€” not white or neon). The glow `div` is not rendered, and the `mousemove` listener is not attached, when `window.matchMedia('(hover: none)').matches` or `window.matchMedia('(prefers-reduced-motion: reduce)').matches` returns `true`.
- [ ] The Magnetic Tag Repulsion effect activates on About section tags when the cursor is within `60px` of a tag's center. The tag shifts via `transform: translate()` in the direction opposite to the cursor, capped at `6px` in any direction. On `mouseleave` or when cursor distance exceeds `60px`, the transform resets to `translate(0, 0)` with an elastic CSS transition. The effect is entirely absent when `(hover: none)` or `(prefers-reduced-motion: reduce)` is detected.
- [ ] All `mousemove` listeners (cursor glow, magnetic tags) are throttled via a `requestAnimationFrame` gate (`rafPending` flag pattern). No listener callback executes more than once per display frame. All `getBoundingClientRect()` reads occur in a single pass before any `style.transform` writes in the same `rAF` callback.
- [ ] Every `useEffect` that registers a `mousemove` listener, a `requestAnimationFrame` loop, or an `IntersectionObserver` returns a cleanup function that calls `removeEventListener`, `cancelAnimationFrame`, and/or `observer.disconnect()` respectively. No listeners or observers persist after the component unmounts.
- [ ] A Lighthouse performance audit run on the page with all effects active scores 90 or above on Performance. The CLS score is `0.00` â€” no layout shift is introduced by any effect in this milestone. All animation-driving CSS properties are limited to `transform` and `opacity`.
- [ ] On a touch device (or a browser emulating `hover: none`), the cursor glow `div` is absent from the DOM, the magnetic repulsion listeners are not attached, and the scroll reveals still function correctly. The page is fully usable and visually complete without any motion effects active.
